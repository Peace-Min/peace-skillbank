# EMPTY_CATCH_BLOCK — 빈 catch 블록 (empty catch)

- **건수**: 89  |  **심각도**: 높음  |  **트랙**: C
- **Sparrow 설명**: `catch` 블록이 비어 있거나 예외를 실질적으로 처리하지 않고 **삼킨다**(주석만 있거나 아무 문장 없음). 예외가 발생해도 로깅·복구·재던지기 없이 무시되어 오류가 은폐된다.

## 지켜야 할 규칙 (무엇을 왜 검출)
예외를 조용히 삼키면 **결함이 은폐**되어 잘못된 상태로 실행이 계속된다. 방산 신뢰성시험에서 예외처리 누락은 핵심 지적 사항 — 오류가 로그에도 남지 않으면 사후 원인분석(사고조사)이 불가능하다. "정상적으로 무시해도 되는 예외"조차 **의도를 코드/주석으로 남겨야** 위양성이 아니라 설계로 인정된다.

## 표준 매핑 (교차참조)
- CWE: **CWE-390** (Detection of Error Condition Without Action); 연관 **CWE-391**(Unchecked Error Condition)
- 무기체계 보안약점 점검 목록: 미매핑 (187 확보 시 "오류상황 대응 부재/부적절한 예외처리" 계열)
- CERT-C/행안부/OWASP: 행안부 "오류상황 대응 부재"; CERT ERR00-J

## 진성 판별 기준
- `catch { }` 또는 `catch (Exception) { }` 본문이 **비어 있음**(또는 주석만).
- 삼킨 예외에 대한 **로깅·상태복구·상위 전파·사용자 통지 중 아무것도 없음**.
- 특히 `catch (Exception)`/`catch` 광범위 포착과 결합되면 위험 가중(→ [[OVERLY_BROAD_CATCH]] 병행 검토).

## 흔한 위양성 패턴
- **의도된 무시가 문서화된 경우**: 예로 `int.TryParse` 대체 목적의 협소한 `catch (FormatException) { /* 기본값 사용 */ }` — 단, 빈 본문이면 여전히 지적 대상이므로 **최소한 주석·기본값 처리**를 남겨야 사유서가 성립.
- Dispose/Cleanup 경로에서 이미 실패한 자원 해제 예외 무시(관례) — 그래도 로깅 권장.
- 대개 EMPTY_CATCH 는 위양성이 드물다. "정말 무시가 옳다"도 **의도 주석 + 좁은 예외형**으로 바꾸는 것이 표준.

## 수정 패턴 (C# 예시)
```csharp
// Before — 삼킴
try { DoWork(); }
catch { }                       // 오류 은폐

// After (A) 로깅 후 정책에 맞게 (재던지기/기본값)
try { DoWork(); }
catch (Exception ex)
{
    _log.Error("DoWork 실패", ex);   // 프로젝트 로깅 사용
    throw;                            // 복구 불가면 전파
}

// After (B) 예외형을 좁히고 의도 명시 (무시가 정말 정상일 때)
try { file.Delete(); }
catch (IOException ex)
{
    // 이미 삭제됨/잠금 — 정리 실패는 무시해도 안전
    _log.Debug("임시파일 삭제 실패(무시)", ex);
}
```
- 기본 방침: **좁은 예외형 + 로깅**. 복구 가능하면 복구, 아니면 `throw;`(스택 보존, `throw ex;` 금지). 무시가 정당하면 **좁힌 예외형 + 사유 주석 + 최소 로깅**.

## 검증 확인 조건 (G2)
- 빌드 통과.
- Sparrow 재분석 시 해당 catch 의 EMPTY_CATCH_BLOCK 소멸.
- 신규 검출 0 — 특히 로깅 호출이 새 UNCHECKED_NULL(로거 null 등) 이나 OVERLY_BROAD_CATCH 를 만들지 않는지.
