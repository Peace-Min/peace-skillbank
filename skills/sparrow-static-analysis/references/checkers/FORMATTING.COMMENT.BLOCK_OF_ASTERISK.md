# FORMATTING.COMMENT.BLOCK_OF_ASTERISK — 주석 별표(*) 블록 사용

- **건수**: (실측 시 변동)  |  **심각도**: 낮음  |  **트랙**: B
- **Sparrow 설명**: 주석문장을 감싸는 별표(*) 블록을 사용하지 말아야 합니다.

> **대부분 Track A(dotnet format/SparrowSyntaxFix) 또는 Track B(SparrowCommentFix/dotnet format whitespace)가 자동 처리한다. 이 가이드는 도구가 못 고친 잔여를 LLM이 처리할 때 사용**한다.

## 지켜야 할 규칙 (무엇을 왜 검출)
여러 줄 주석을 각 줄 앞에 별표(`*`)를 붙여 **별표 블록으로 서식화하지 말아야** 한다. **보안 결함이 아니라 스타일·가독성·표준(MSDN C# 코딩 규칙) 준수** 항목이다. 별표 블록은 편집 시 정렬 유지 부담이 크고 C# 관례가 아니다. Sparrow는 `/* ... * ... * ... */`처럼 내부 줄마다 `*`를 붙인 주석을 검출한다.

## 표준 매핑 (교차참조)
- MSDN C#:2015 코딩 규칙 **3.5** — 서식이 지정된 별표 블록으로 주석을 묶지 않습니다.
- C# Coding conventions : 2023 — 서식이 지정된 별표 블록으로 주석을 묶지 않습니다.
- CWE: 스타일 항목이라 직접 매핑 없음(광의 **CWE-1078** 계열).
- 무기체계 보안약점 점검 목록: **187 항목 100% 적용**; 개별 항목번호는 Sparrow 체커↔표준 매핑 추출 후 기입(대기).

## 결함 판별 기준
- 블록 주석(`/* */`) 내부 줄들이 선두 `*`로 정렬된 별표 블록 형태다.
- 예: `/* BadCase1` / ` * BadCase1` / ` * BadCase1 */` — 결함.

## 이렇게 보여도 넘기지 말 것
스타일 항목이라도 **전건 수정**한다. 별표 블록이 보기 좋게 정렬돼 있어도 표준 형태로 바꾼다.
- **XML 문서 주석**(`///`)이나 별표가 실제 내용(ASCII 아트, 수식의 곱셈 `*` 등)인 경우는 이 규칙의 "장식용 별표 블록"과 다르므로, 내용을 훼손하지 말고 그 근거를 판정에 기록한다(스킵이 아니라 판정 근거).
- 저작권/헤더 배너처럼 조직 표준 서식이 별표 블록을 요구하는 경우가 있으면, 프로젝트 규약을 확인한 뒤 처리한다(문맥 확보).
- 별표 블록이 관례처럼 보여도 표준은 별표 없는 여러 줄 주석이다 — 근거 확인 후 수정한다.

## 수정 패턴 (C# 예시)
```csharp
// Before — 각 줄 선두 별표 블록
/* BadCase1
 * BadCase1
 * BadCase1 */
Console.WriteLine("BadCase1");

// After — 별표 없이 들여쓴 여러 줄 주석
/* GoodCase1
   GoodCase1
   GoodCase1 */
Console.WriteLine("GoodCase1");
```

## 검증 확인 조건 (G2)
- 빌드 통과(net472 / C# 7.3) — 주석 변경은 의미에 영향 없음.
- 재분석 시 해당 라인 FORMATTING.COMMENT.BLOCK_OF_ASTERISK 소멸.
- 신규 검출 0 — 주석 내용 텍스트를 손실 없이 옮겼는지 확인.
