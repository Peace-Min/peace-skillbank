# FORMATTING.COMMENT.LOWERCASE_FIRST_LETTER — 주석 소문자 시작

- **건수**: (실측 시 변동)  |  **심각도**: 낮음  |  **트랙**: B
- **Sparrow 설명**: 주석 문장은 대문자로 시작해야 합니다.

> **대부분 Track A(dotnet format/SparrowSyntaxFix) 또는 Track B(SparrowCommentFix/dotnet format whitespace)가 자동 처리한다. 이 가이드는 도구가 못 고친 잔여를 LLM이 처리할 때 사용**한다.

## 지켜야 할 규칙 (무엇을 왜 검출)
주석 문장은 **대문자로 시작**해야 한다. **보안 결함이 아니라 스타일·가독성·표준(MSDN C# 코딩 규칙) 준수** 항목이다. 첫 글자를 대문자로 통일하면 주석이 문장임이 드러나고 문서 스타일이 일관된다. Sparrow는 소문자로 시작하는 주석 문장을 검출한다.

## 표준 매핑 (교차참조)
- MSDN C#:2015 코딩 규칙 **3.2** — 주석 텍스트는 대문자로 시작합니다.
- C# Coding conventions : 2023 — 주석 텍스트는 대문자로 시작합니다.
- CWE: 스타일 항목이라 직접 매핑 없음(광의 **CWE-1078** 계열).
- 무기체계 보안약점 점검 목록: **187 항목 100% 적용**; 개별 항목번호는 Sparrow 체커↔표준 매핑 추출 후 기입(대기).

## 진성 판별 기준
- 주석문의 첫 알파벳 문자가 소문자다.
- 예: `// badCase1` — 진성. `// GoodCase1.` — 정상.

## 이렇게 보여도 넘기지 말 것
스타일 항목이라도 **전건 수정**한다. 소문자 시작이 의도처럼 보여도 표준(대문자 시작)으로 고친다.
- 첫 토큰이 **식별자/코드 심볼**(예: `// camelCaseVar 는 지역 변수`, `// iPhone`, `// gRPC`)이라 대문자화하면 의미가 바뀌는 경우에는, 실제 심볼 이름을 훼손하지 말고 서술을 앞에 붙이거나 문장을 대문자로 시작하도록 재구성한다. 원 심볼 표기를 그대로 둬야 하면 그 근거를 판정에 기록한다(스킵이 아니라 판정 근거).
- 한글 주석은 대소문자 개념이 없으므로 이 체커 대상이 아니다(주로 영문 주석에 적용).
- 소문자 시작이 관례처럼 보여도 표준은 대문자 시작이다 — 근거 확인 후 수정한다.

## 수정 패턴 (C# 예시)
```csharp
// Before
// badCase1
Console.WriteLine("BadCase1");

// After
// BadCase1.
Console.WriteLine("BadCase1");
```

## 검증 확인 조건 (G2)
- 빌드 통과(net472 / C# 7.3) — 주석 변경은 의미에 영향 없음.
- 재분석 시 해당 라인 FORMATTING.COMMENT.LOWERCASE_FIRST_LETTER 소멸.
- 신규 검출 0 — 심볼 이름을 훼손하지 않았는지 확인.
