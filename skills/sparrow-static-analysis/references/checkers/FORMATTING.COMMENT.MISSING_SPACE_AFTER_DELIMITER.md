# FORMATTING.COMMENT.MISSING_SPACE_AFTER_DELIMITER — 주석 구분자 뒤 공백 누락

- **건수**: (실측 시 변동)  |  **심각도**: 낮음  |  **트랙**: B
- **Sparrow 설명**: 주석문장과 주석구분자(//) 사이에 공백이 있어야 합니다.

> **대부분 Track A(SparrowSyntaxFix/SparrowSyntaxFix) 또는 Track B(SparrowCommentFix/SparrowCommentFix layout)가 자동 처리한다. 이 가이드는 도구가 못 고친 잔여를 LLM이 처리할 때 사용**한다.

## 지켜야 할 규칙 (무엇을 왜 검출)
주석 구분 기호(`//`)와 주석 텍스트 사이에는 **공백을 하나 이상** 넣어야 한다. **보안 결함이 아니라 스타일·가독성·표준(MSDN C# 코딩 규칙) 준수** 항목이다. `//BadCase1`처럼 붙여 쓰면 판독성이 떨어지고 팀 스타일과 어긋난다. Sparrow는 `//` 바로 뒤에 공백 없이 텍스트가 붙은 주석을 검출한다.

## 표준 매핑 (교차참조)
- MSDN C#:2015 코딩 규칙 **3.4** — 주석 구분 기호(//)와 주석 텍스트 사이에 공백을 하나 삽입합니다.
- C# Coding conventions : 2023 — 주석 구분 기호(//)와 주석 텍스트 사이에 공백을 하나 삽입합니다.
- CWE: 스타일 항목이라 직접 매핑 없음(광의 **CWE-1078** 계열).
- 무기체계 보안약점 점검 목록: **187 항목 100% 적용**; 개별 항목번호는 Sparrow 체커↔표준 매핑 추출 후 기입(대기).

## 결함 판별 기준
- `//` 다음에 공백 없이 즉시 텍스트가 온다(예: `//BadCase1`).
- 정상: `// GoodCase1`.

## 이렇게 보여도 넘기지 말 것
스타일 항목이라도 **전건 수정**한다. 붙여 쓴 주석이 의도처럼 보여도 공백을 하나 넣는다.
- **주석 처리된 코드**(`//return x;`)도 `//` 뒤 공백 규칙 대상이면 `// return x;`로 공백을 넣되, 코드 토큰 자체는 바꾸지 않는다.
- `///`(XML 문서 주석) 뒤 규칙, 구분선 주석(`//////////`)처럼 특수 형태는 도구/규칙 정의에 따라 처리하고, 서술 텍스트가 붙어 있으면 첫 텍스트 앞에 공백을 넣는다.
- 공백 없는 형태가 관례처럼 보여도 표준은 `// ` 뒤 공백이다 — 근거 확인 후 수정한다.

## 수정 패턴 (C# 예시)
```csharp
// Before
//BadCase1
Console.WriteLine("BadCase1");

// After
// GoodCase1
Console.WriteLine("GoodCase1");
```

## 검증 확인 조건 (G2)
- 빌드 통과(net472 / C# 7.3) — 주석 변경은 의미에 영향 없음.
- 재분석 시 해당 라인 FORMATTING.COMMENT.MISSING_SPACE_AFTER_DELIMITER 소멸.
- 신규 검출 0 — 공백만 추가하고 코드 토큰을 건드리지 않았는지 확인.
