# MISSING_PARENTHESIS_IN_EXPRESSION — 모호한 수식 우선순위

- **건수**: (실측 시 변동)  |  **심각도**: 낮음  |  **트랙**: A
- **Sparrow 설명**: 표현식 안에서 명확한 절을 만들기 위해 괄호를 사용해야 합니다.

> **대부분 Track A(SparrowSyntaxFix/SparrowSyntaxFix) 또는 Track B(SparrowCommentFix/SparrowCommentFix layout)가 자동 처리한다. 이 가이드는 도구가 못 고친 잔여를 LLM이 처리할 때 사용**한다.

## 지켜야 할 규칙 (무엇을 왜 검출)
서로 다른 우선순위의 연산자가 한 표현식에 섞여 있을 때, 연산자 우선순위에만 의존하지 말고 **괄호로 의도한 결합 순서를 명시**해야 한다. **보안 결함이 아니라 가독성·오독 방지·표준 준수** 항목이다. `a && b || c`, `a + b << c`, `x & y == z`처럼 &&/||, 비트/시프트/비교 혼용은 사람이 우선순위를 오독하기 쉬워 잠재적 논리 버그의 온상이 된다. 괄호는 컴파일 결과를 바꾸지 않으면서 의도를 드러낸다. Sparrow는 우선순위가 다른 이항 연산자가 괄호 없이 혼합된 수식을 검출한다. (Roslyn IDE0048과 동일 취지.)

## 표준 매핑 (교차참조)
- **Rule 원문 미확보 — 확보 시 레퍼런스 보강.** 아래는 도메인 지식 기반 참고 매핑이다.
- Roslyn 스타일 규칙 **IDE0048**(Add parentheses for clarity) / 방위사업청 시큐어코딩 가독성 지침의 수식 괄호 명시.
- CWE: 스타일 항목이라 직접 매핑 없음(광의 **CWE-1078** 계열; 오독 유발 시 **CWE-783** Operator Precedence Logic Error와 인접).
- 무기체계 보안약점 점검 목록: **187 항목 100% 적용**; 개별 항목번호는 Sparrow 체커↔표준 매핑 추출 후 기입(대기).

## 결함 판별 기준
- 한 표현식에 **우선순위가 서로 다른 이항 연산자**가 괄호 없이 섞여 있다: `&&`와 `||`, 산술과 시프트/비트, 비교와 비트(`&`,`|`,`^`), `??`와 비교 등.
- 대표 예: `if (a || b && c)`, `int r = a + b << 2;`, `bool t = x & mask == 0;`

## 이렇게 보여도 넘기지 말 것
스타일 항목이라도 **전건 수정**한다. "우선순위를 알고 썼다"처럼 보여도 괄호를 넣어 의도를 명시한다.
- 괄호 추가는 **의미를 바꾸지 않는 그룹핑**이어야 한다 — 실제 우선순위대로 괄호를 씌운다(예: `a || b && c` → `a || (b && c)`). 잘못된 그룹핑으로 로직을 바꾸지 않도록, 현재 우선순위를 먼저 확인한 뒤 그대로 괄호로 감싼다.
- 이미 논리 버그가 의심되면(괄호 위치가 의도와 다를 소지) 임의로 재해석하지 말고, 현행 우선순위를 보존하는 괄호를 넣은 뒤 그 사실을 판정에 기록한다(문맥 확보 후 로직 검토).
- 의도적으로 괄호를 뺀 것처럼 보여도 표준은 명시 괄호다 — 근거 확인 후 수정한다.

## 수정 패턴 (C# 예시)
```csharp
// Before — 우선순위에만 의존
if (a || b && c) { }
int r = a + b << 2;
bool t = x & mask == 0;

// After — 실제 우선순위대로 괄호 명시(의미 불변)
if (a || (b && c)) { }
int r = (a + b) << 2;
bool t = x & (mask == 0 ? 1 : 0); // 원래 의도가 비교면 아래처럼 분리
// 또는 의도가 (x & mask) == 0 이면:
bool t2 = (x & mask) == 0;
```

## 검증 확인 조건 (G2)
- 빌드 통과(net472 / C# 7.3).
- 재분석 시 해당 라인 MISSING_PARENTHESIS_IN_EXPRESSION 소멸.
- 신규 검출 0 — **괄호 추가가 기존 우선순위/평가 결과를 바꾸지 않았는지**(동치 변환) 반드시 확인.
