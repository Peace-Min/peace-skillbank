# PRACTICE.ARRAY_DECLARATION.COMPLICATED_SYNTAX — 복잡한 배열 초기화 문법

- **건수**: 1  |  **심각도**: 낮음  |  **트랙**: C
- **Sparrow 설명**: 선언문에서 배열을 초기화할 때, 간략한 문법을 사용해야 합니다.

## 지켜야 할 규칙 (무엇을 왜 검출)
선언과 동시에 배열을 초기화할 때 `new T[] { ... }` 처럼 **장황한 문법**을 쓰면 타입 정보가 중복되어 가독성이 떨어진다. C#은 선언문에서 컬렉션 초기화식만으로도 요소 타입을 추론하므로 **간략한 문법**을 사용하는 것이 권장된다. 직접적 보안결함은 아니지만(낮음), 방산 코드의 가독성·검토용이성 기준에서 명료한 표준 구문이 권장된다. 의도된 형태일 수 있어 판단이 필요하므로 트랙 C이며, 1건이므로 검토 후 수동 정리한다.

## 표준 매핑 (교차참조)
- CWE: 직접 매핑 없음(스타일/유지보수 권고). 광의로 **CWE-1078**(Inappropriate Source Code Style) 계열
- 무기체계 보안약점 점검 목록: **187 항목 100% 적용(설정 확정)**; 개별 항목번호는 Sparrow 체커↔표준 매핑 추출 후 기입(대기) — 단 스타일 계열이라 실매핑 대상이 아닐 수 있음
- .NET Framework Design Guideline: 선언 초기화 시 간결한 컬렉션 초기화식 사용

## 진성 판별 기준
- **선언과 동시에 초기화**하는 배열이 `new T[] { ... }`처럼 타입을 중복 명시하는 장황한 문법을 사용한다.
- 동일 의미를 더 간략한 표준 문법(`= { ... }` 또는 `new[] { ... }`)으로 표현 가능하다.

## 이렇게 보여도 넘기지 말 것
아래처럼 "축약 불가"로 보여도 **스킵하지 말고**, 가능한 형태로 정리한다(전건 수정).
- 선언과 초기화가 분리된 경우(대입만 하는 위치)에는 `= { ... }` 축약이 불가하지만 `new[] { ... }` 로 타입 중복을 줄일 수 있는지 확인해 정리한다. 정말 축약 여지가 없으면 그 근거를 판정에 남긴다(스킵이 아니라 근거 기록).
- 요소 타입 추론이 애매해(혼합 리터럴 등) 명시적 `new T[]`가 오히려 명료한 경우도 — 의미를 바꾸지 않는 선에서 정리 가능한지 확인하고, 불가하면 근거를 남긴다.

## 수정 패턴 (C# 예시)
```csharp
// Before — 장황
int[] a = new int[] { 1, 2, 3 };
string[] s = new string[] { "x", "y" };

// After — 간략
int[] a = { 1, 2, 3 };          // 선언+초기화 시 타입 생략 가능
var b = new[] { 1, 2, 3 };       // var 로 받을 땐 new[] 형태(타입 추론)
```
- 원칙: **선언과 동시에 초기화**할 땐 `= { ... }`로, `var`/대입 위치에선 `new[] { ... }`로 축약한다. 요소 타입·개수·값의 의미가 바뀌지 않도록 그대로 보존한다.
