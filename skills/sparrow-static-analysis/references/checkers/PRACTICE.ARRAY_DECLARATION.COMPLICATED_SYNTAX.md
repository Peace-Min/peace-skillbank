# PRACTICE.ARRAY_DECLARATION.COMPLICATED_SYNTAX — 복잡한 배열 선언 구문 (readability)

- **건수**: 1  |  **심각도**: 낮음  |  **트랙**: C
- **Sparrow 설명**: 배열 선언이 **불필요하게 복잡한 구문**(중첩·혼동되기 쉬운 형태, 예: C 스타일 배열 크기 표기·과한 다차원/가변배열 혼용)을 사용한다. 가독성·오해 소지를 줄이도록 명료한 표준 구문으로 재작성을 권고. 소량(1건)이라 **수동 처리**.

## 지켜야 할 규칙 (무엇을 왜 검출)
읽기 어려운 배열 선언은 유지보수 중 오해(차원·크기 착각)를 유발해 인덱스 오류·초기화 실수의 간접 원인이 될 수 있다. 직접적 보안결함은 아니나(낮음), 방산 코드 가독성·검토용이성 기준에서 명료한 표준 구문이 권장된다. 판단(의도된 형태인지)이 필요해 트랙 C이지만, 1건이므로 검토 후 수동 정리.

## 표준 매핑 (교차참조)
- CWE: 직접 매핑 없음(가독성/유지보수 권고). 광의로 **CWE-1078**(Inappropriate Source Code Style) 계열
- 무기체계 보안약점 점검 목록: 미매핑 (스타일 — 187 대상 아닐 가능성 높음)
- CERT-C/행안부/OWASP: 코딩 표준(명료성) 일반 원칙

## 진성 판별 기준
- 배열 선언이 표준 C# 구문에서 벗어나거나 차원/크기 표기가 혼동을 준다.
- 동일 의미를 더 단순한 표준 구문으로 표현 가능하다.

## 흔한 위양성 패턴
- 성능/상호운용(예: `unsafe`/`fixed`, P/Invoke 레이아웃) 때문에 특정 형태가 **의도적으로 필요**한 경우 → 사유서.
- 이미 명료하며 분석기의 스타일 취향 차이일 뿐인 경우.

## 수정 패턴 (C# 예시)
```csharp
// Before — 혼동되는 선언 (예시)
int arr[] = ...;                 // C 스타일 크기표기/비표준 형태
// 다차원·가변 혼용으로 의미가 모호한 선언

// After — 표준 C# 배열 구문으로 명료화
int[] arr = new int[n];          // 1차원
int[,] grid = new int[rows, cols];   // 다차원(직사각)
int[][] jagged = new int[rows][];    // 가변(재그드) — 의도가 명확할 때만
```
- 실제 코드 형태를 확인해 **의미 보존**하며 표준 구문으로 정리(차원/크기 의미가 바뀌지 않도록).
