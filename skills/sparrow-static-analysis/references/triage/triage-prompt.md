# Track C 실제 수정 요청 프롬프트

> 이 파일은 `Run-Triage.ps1 prepare`와 Sparrow Helper GUI가 Track C 요청 md를 만들 때 사용하는 템플릿이다.
> `[체커 가이드]`에는 해당 체커 가이드 전문이, `[검출 항목]`에는 Sparrow XLS에서 추출한 항목 md 전문이 들어간다.
> 생성된 `requests/<체커>/<ID>_<체커>.md` 하나가 LLM/사람에게 전달되는 self-contained 작업 요청이다.
>
> 자리표시자(조립 직전 리터럴 치환. Run-Triage.ps1 과 TriagePreparer.cs 가 동일 문자열을 넣는다):
> - `{{GUIDE}}` / `{{ITEM}}` — 체커 가이드 전문 / 항목 md 전문.
> - `{{EVIDENCE_FIELD}}` — 처리 상태의 근거 필드. **등록** 체커는 `근거:` 1줄(가이드 요약),
>   **미등록** 체커는 `근거(인용):` + `근거(코드):` 2줄(요약 대신 인용 강제 — 없는 룰을 지어내지 못하게).
> - `{{UNREGISTERED_CONSTRAINT}}` — 미등록 체커에만 삽입되는 인용-근거 제약 문장. 등록 체커는 빈 문자열이라
>   바로 뒤 문장("아래 형식을 유지한다.")에 이어 붙는다.

---

## 역할

당신은 Sparrow 정적분석 검출 1건을 실제 코드 수정으로 연결하는 작업자다.
아래 **[체커 가이드]**와 **[검출 항목]**을 읽고, 접근 가능한 실제 소스 파일에서 결함을 제거한다.

> **[필수] 스킵·제외·"해당 없음"·"문제 없음" 결론 금지. 산출물 없는 종료는 실패로 간주합니다.**

이 프로젝트는 전건 수정 정책이다. "코드상 문제 없어 보임", "false-positive로 보임" 같은 이유만으로 방치하지 않는다. 정말 수정할 수 없으면 `문맥 필요`로 두고 필요한 파일, 심볼, 호출부, 소유권 정보를 구체적으로 적는다.

## 작업 규칙

1. 실제 소스 파일 접근과 수정 권한이 있으면 파일을 열어 직접 수정한다.
2. 파일 접근 권한이 없거나 현재 문맥만으로 실제 수정이 불가능하면 최소 patch 또는 Before/After를 작성하고 상태를 `패치 제안`으로 한다. 접근 불가는 작업을 건너뛸 사유가 아니다.
3. `[검출 항목]`의 `TARGET LINE ... ANCHOR`는 수정 기준점이다. 결함 제거에 필요한 최소 인접 범위까지 수정하되, 무관한 주변 코드는 임의 수정하지 않는다.
4. C# 코드는 .NET Framework 4.7.2 / C# 7.3 문법으로 작성한다.
   - `using (...) { }` 블록 문법만 사용한다.
   - using 선언, switch 식, nullable reference type 문법은 사용하지 않는다.
5. 조용한 무시, 빈 catch, 의미 없는 null 병합처럼 결함을 숨기는 수정은 금지한다.
6. 출력은 Markdown 처리 결과로 작성한다. JSON 설정 파일은 작성하지 않는다.

## 출력 형식

{{UNREGISTERED_CONSTRAINT}}아래 형식을 유지한다.

```markdown
# <체커키> 처리 결과

## 대상
- ID:
- 파일:
- 라인:
- 체커:

## 처리 상태
- 상태: 수정 완료 | 패치 제안 | 문맥 필요 | 판정 근거 기록(수정 불요)
{{EVIDENCE_FIELD}}
- 비결함 근거(a 코드 인용): <`판정 근거 기록(수정 불요)`일 때만. 아니면 "해당 없음">
- 비결함 근거(b 잔여 리스크): <`판정 근거 기록(수정 불요)`일 때만. 아니면 "해당 없음">
- 비결함 근거(c 소유 계약 출처): <`판정 근거 기록(수정 불요)`일 때만. 아니면 "해당 없음">

## 변경 내용
<실제 수정한 파일과 핵심 변경 사항. 직접 수정하지 못했다면 적용할 patch 또는 Before/After>

## 추가 필요 문맥
- <없으면 "없음">

## 검증 (실행은 사용자가 수행 — 결과를 추측해 적지 말 것)
- 빌드 영향 범위: <이 수정이 영향을 주는 프로젝트/파일>
- 재분석 시 소멸해야 할 검출: <체커키>@<파일>:<라인>
```

- **상태 결정 규칙(배타)**: 실제 파일을 고쳤으면 `수정 완료`. 고치지 못했더라도 **Before/After 또는 patch를 제시했으면 `패치 제안`**(둘 다 쓰면서 `문맥 필요`로 적지 말 것). patch조차 낼 수 없을 때만 `문맥 필요`. 위 배너의 "'문제 없음' 결론 금지"는 아래 증빙 3요건을 **모두** 충족한 `판정 근거 기록(수정 불요)`에만 예외를 둡니다.
- `판정 근거 기록(수정 불요)`은 (a) 왜 결함이 아닌지 **코드 인용**, (b) Sparrow 검출이 가리키는 잔여 리스크, (c) 소유 관계를 근거로 들 경우 그 소유 계약의 출처(코드 위치/문서)를 **모두** 적을 때만 사용할 수 있습니다. 추정만으로는 사용 금지. 이 3요건은 위 `## 처리 상태`의 `비결함 근거(a/b/c)` 칸에 적고, 템플릿 밖에 임의 항목을 만들지 마십시오.
